A.4 Record Number 自增方案

Design Overview

采用“按 app 维度单行计数器 + 插入阶段自动分配 + 唯一约束兜底”的组合策略，实现统一发号。核心原则是：业务侧创建记录默认不传 record_number，由数据库层在写入时按 app_id 统一分配，保证所有写入口规则一致。

场景行为保证
应用层不传 record_number触发器调用 next_app_record_number(app_id) 自动发号。同 app 单调递增。
导入/修复显式传 record_number保留传入编号，并同步 counter 到 GREATEST(last_number, NEW.record_number)。后续发号不会回退或冲突。
并发创建同一 app_id 竞争同一计数行，由 PostgreSQL 行锁串行化。无重复号。
最终兜底UNIQUE(app_id, record_number) 约束兜底。阻断重复写入。
  1. 应用层执行 INSERT（默认不传 record_number）。
  2. BEFORE INSERT 触发器触发并分配编号。
  3. 写入 records.record_number 后提交事务。
  4. 唯一索引再次兜底，确保无重复。
边界项语义
硬删除后编号复用不会复用，counter 只增不减。
事务回滚语义counter 更新与 records 插入在同一事务中，失败会一起回滚。
编号连续性保证单调递增；导入/修复场景允许跳号。
与 version 的关系record_number 发号不依赖乐观锁 version。

设计要点总结

  • 编号空间按 app 隔离，避免跨应用互相影响。
  • 并发下同一 app 的发号按计数行串行化，保证不重复。
  • 导入/修复支持显式编号，并同步提升计数器，防止后续回退。
  • 应用层保持无状态发号，数据库层统一收口与兜底。

Appendix B: Data Structure Specifications

B.1 Field Definition Specification

apps.fields

对应 Kintone 的 Form Fields。定义数据的结构与约束。

存储格式 (JSON)
[
  {
    "id": "fld_1f3e21cd00",
    "code": "Number",
    "type": "number",
    "label": "数值输入",
    "required": false,
    "unique": false,
    "noLabel": false,
    "defaultValue": "",
    "options": []
  },
  {
    "id": "fld_07d647171e",
    "code": "Text",
    "type": "input",
    "label": "单行输入",
    "maxLength": 64
  },
  {
    "id": "fld_table_1",
    "code": "Table1",
    "type": "table",
    "fields": [{ "code": "col1", "type": "input", "label": "列1" }]
  }
]
关键属性表
属性类型说明适用类型
idstringImmutable ID. 内部唯一标识。3.3全部
codestringMutable Code. 应用内唯一。全部
typestringImmutable Type.全部
labelstring显示名称。3.3全部
requiredboolean是否必填。大部分
uniqueboolean是否判重。部分
defaultValueany默认值。大部分
optionsarray选项列表。3.3select/radio/checkbox
fieldsarray子表列定义。table
允许的 type 枚举（Latest）

input / number / textarea / datepicker / timepicker / select / checkbox / radio / attachment / richtext / plate-richtext / table / lookup / user-select / divider / label / step-separator

结构与业务约束
  • 仅 table 可携带 fields。
  • table 不允许 options。
  • 仅 select/radio/checkbox 允许 options，且 id 在同字段内唯一。3.3
  • code 必须匹配 ^[A-Za-z_][A-Za-z0-9_]*$，并在顶层与子表列中全局唯一。
  • 同一 id 的 type 不可变更。
  • 输入 id 缺失或无 fld_ 前缀时，写入前会自动规范化为新 ID。

B.2 Default View Configuration

apps.view_config

定义默认表单的 Layout。

核心原则: Layout 必须使用 Field ID 来引用字段。

存储格式 (JSON)
{
  "layout": [
    {
      "type": "ROW",
      "id": "row_1",
      "fields": [
        { "id": "fld_1f3e21cd00", "width": 180, "height": 36 },
        { "id": "fld_07d647171e", "width": 220, "height": 36 }
      ]
    },
    {
      "type": "SUBTABLE",
      "id": "fld_table_1",
      "fields": [{ "id": "fld_col1", "width": 150, "height": 36 }]
    },
    {
      "type": "SPLITTER",
      "id": "split_1",
      "leftLines": [{ "fields": [...] }],
      "rightLines": [{ "fields": [...] }],
      "splitRatio": 0.5
    }
  ],
  "layoutMode": "auto"
}
Schema 结构约束
层级字段约束
Top-levellayout必填，类型为 LayoutItem[]，对象为 strict 模式。
ROWfields必填，元素为 LayoutField。
SUBTABLEid / fieldsid 必填（子表字段 ID），fields 必填。
SPLITTERleftLines/rightLines/splitRatio左右分栏必填；splitRatio 可选且 0 <= ratio <= 1。
LayoutFieldid/width/height/codeid 必填；width/height 可选且 >= 0；code 为兼容输入字段，落库前清理。
API 返回策略

后端 API 返回时，会自动注入 code：

API Response
{
  "layout": [
    {
      "code": "Number",
      "id": "fld_1f3e21cd00",
      "width": 180
    }
  ]
}
