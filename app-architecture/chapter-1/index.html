1.2 字段定义与视图同步

Field & View Sync

注

详细的 apps.fields JSON 结构与字段属性列表，请参考 Appendix B.1: Field Definition Specification。

注

详细的 apps.view_config (Default View) JSON 结构，请参考 Appendix B.2: Default View Configuration。

apps.fields / view_config 校验要点（来自 apps-design）

  • type 枚举: input / number / textarea / datepicker / timepicker / select / checkbox / radio / attachment / richtext / plate-richtext / table / lookup / user-select / divider / label / step-separator。
  • Schema 约束: 仅 table 可含 fields；table 不允许 options；仅 select/radio/checkbox 允许 options。
  • 业务约束: code 格式 ^[A-Za-z_][A-Za-z0-9_]*$，且顶层与子表列全局唯一。
  • 不可变约束: 同一 field id 的 type 不允许变更（子表列同样适用）。
  • ID 规范化: 缺失或非 fld_ 前缀，会在写入时重置为新 ID。

Schema (fields)与 Default View (view_config) 之间严格遵循 单向同步 (One-Way Sync)。

关键约束: 字段类型不可变 (Immutable Type)

一旦字段被创建，其 type 属性（如 text, number）永久不可变更。

  • 系统不存在 Update Type 这一原子操作。
  • 若用户需要“修改类型”，必须手动执行 Delete Old Field + Create New Field。
  • 因此，旧字段的数据会被视为删除（残留），新字段的数据为空。
  • Trigger: 任何对字段的增删改 (Schema Change)。
  • Effect: 自动计算并更新视图配置 (View/Layout Update)。
  • Constraint: 视图层的修改（列宽、排序）绝不逆向影响 Schema。

1.3 专用视图策略

Specialized Views

对于存储在 app_views 中的日历、看板等专用视图，它们不需要包含所有字段，而是依赖特定的 关键字段 (Key Fields) 配置。

核心原则: 所有的配置项必须存储 Field ID (fld_xxxxx)，严禁存储 Field Code。

View TypeConfig 示例 (JSONB)依赖关系
Calendar
{ "startDateFieldId": "fld_a" }强依赖特定的日期字段 ID。
Kanban
{ "groupFieldId": "fld_s" }强依赖分组字段 ID。

健壮性与恢复 (Robustness & Recovery)

提问: 如果用户删除了日历视图依赖的"开始时间"字段，会发生什么？

回答: 系统不会崩溃，只会降级。

  • State: View Config 指向了一个不存在的 fld_id。
  • Behavior (Runtime): 前端组件检测到关键配置丢失，显示 “配置错误: 缺少日期字段” 的占位提示 (Fallback UI)。
  • Recovery (User Action):用户进入视图设置，重新选择一个新的日期字段即可。

结论: 对于专用视图，字段删除 = 配置失效。这只是一个需用户手动修复的配置状态，而非系统故障。

各类视图的详细配置结构，请参考 Appendix B.3: Specialized View Config。

1.4 复杂组件架构

Complex Component Architecture

针对 子表 (Subtable) 与 分栏 (Splitter) 等复杂嵌套组件，系统采用“职责分离”的架构设计，确保灵活性与性能的平衡。

1.4.1 子表模型 (Subtable Model)

  • Schema: 在 apps.fields 中定义为 type: 'table'，并通过嵌套的 fields 数组定义列。
    优势: 物理上隔离了主表与子表的 Schema 定义，避免命名空间污染。
  • Layout: Default View 中的 SUBTABLE 布局项仅通过 ID 引用表格字段，并在其 fields 数组中独立定义可见列 (Visible Columns)与顺序。
    优势: 实现了 View 与 Schema 的解耦。用户可以随意调整表格列的显示顺序或隐藏某些列，而无需修改底层数据结构。
  • Data: 在 records.values 中存储为对象数组 (Array<Object>)，并以表格字段 ID 为 Key。
    优势: 符合 NoSQL 的标准嵌套范式，查询高效且保证了 1:N 数据的原子性读写。

1.4.2 分栏模型 (Splitter Model)

  • Structure: 定义包含 leftLines 和 rightLines，它们是包含字段的"行 (Lines)"数组。
  • Consistency: 这种 "Layout → Splitter → Lines → Fields" 的类递归结构，有效地为分割线的每一侧创建了独立的布局容器 (Block Context)。
  • Scalability:虽然当前设计严格限制容器内仅能存放字段，但这种嵌套结构为未来扩展（如 Splitter 嵌套 Splitter、嵌入图表）预留了架构空间。
