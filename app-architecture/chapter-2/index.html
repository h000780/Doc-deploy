记录设计

Records 设计

本章节涵盖 records 与 app_record_counters 的联合设计，负责记录数据存储、索引策略与编号分配。

2.1 存储模型

Storage Model

  • Core Tables: records + app_record_counters。
  • Data Payload: values (JSONB)，以 fld_xxxxx 作为稳定 Key。
  • Business Numbering: record_number 在同一 app_id 内单调递增，并由数据库触发器统一分配。
  • Uniqueness Guard: UNIQUE(app_id, record_number) 作为最终兜底，阻止重复号落库。

2.2 读写转换与校验

IO Translation & Validation

为了在后端物理层保持稳定的 fld_xxxxx ID，同时在前端逻辑层提供语义化的 fieldCode 体验，系统采用 "双层网关 (Dual-Gateway)" 架构，并配合 请求级上下文 (Request-Scoped Context) 进行高效转换。

核心架构 (Core Architecture)

  1. Context (上下文): AsyncLocalStorage (CLS)
    • Single Source of Truth: 在 Middleware 层一次性解析 App Schema，构建 Field Map (Code ↔ ID) 并挂载到请求上下文中。
    • Performance: 确保每个请求只解析一次 Schema (O(1) per request)，Pipe 和 Interceptor 直接从 Context 读取映射，无需重复计算。
    • Tip: 在 Service 层或任何需要多次查询 Field 对应关系的场景中，必须 优先从 CLS Context 获取，严禁重复调用解析逻辑。
  2. Input Gateway (写链路): ValidationPipe
    • Translate (Code → ID): 在请求进入 Controller 之前，利用 Context 将 payload 中的 fieldCode 为 fld_id。
    • Validate: 基于 Schema 对输入数据进行严格校验 (Zod)，确保写入 JSONB 的数据类型安全。
  3. Output Gateway (读链路): TransformInterceptor
    • Translate (ID → Code):在响应返回给前端之前，利用 Context 将 DB 中的 fld_id 为 fieldCode。
    • Sanitize (净化): 采用白名单机制，剔除 Schema 中不存在的残留字段 (fld_orphan)，彻底杜绝"僵尸数据"泄露。

实现细节 (Implementation Details)

  • Input Pipe (ValidationPipe):
    • 负责写入前的清洗。
    • 此时 DB 中尚无数据，纯粹是基于 Schema 的映射。如果不符合 Schema，Pipe 层直接拦截。
  • Output Interceptor:
    • 负责读取后净化。
    • 关键逻辑: 遍历 DB 返回的 Raw Data，尝试在 Context 中查找 fld_id 对应的 fieldCode。
    • Orphan Handling: 如果找不到对应的 Code（说明该字段已被删除），则直接忽略该键值对，确保它不会出现在最终响应中。

2.3 索引策略

Indexing

为了覆盖应用隔离、JSONB 查询、全文搜索与访问控制， records 表需要组合索引，而不只是单一的 JSONB GIN：

CREATE INDEX IF NOT EXISTS records_app_id_idx
ON records(app_id);

CREATE INDEX IF NOT EXISTS records_values_gin_idx
ON records USING GIN(values);

CREATE INDEX IF NOT EXISTS idx_records_access
ON records(app_id, created_by);

CREATE INDEX IF NOT EXISTS idx_records_org_unit
ON records(app_id, org_unit_id);

CREATE INDEX IF NOT EXISTS records_deleted_at_idx
ON records(deleted_at);

-- Full-text index can be built after converting search column to generated column
CREATE INDEX IF NOT EXISTS idx_records_search_text
ON records USING GIN(search_text gin_bigm_ops);

-- Final uniqueness constraint for record_number
CREATE UNIQUE INDEX IF NOT EXISTS records_app_record_unique_idx
ON records(app_id, record_number);

2.4 Record Number 分配机制

Allocation Strategy

采用 （独立计数器表）：通过 next_app_record_number(app_id) + BEFORE INSERT 触发器统一发号，应用层默认不传 record_number。

-- Application-layer record creation (without record_number)
INSERT INTO records (app_id, "values", created_at, created_via)
VALUES ($1, $2::jsonb, now(), 'user')
RETURNING *;
  1. 触发器优先检查 NEW.record_number 是否为空；为空则调用发号函数。
  2. 发号函数通过 INSERT ... ON CONFLICT DO UPDATE 原子递增同一 app 的计数行。
  3. 同一 app_id 的并发写入由行锁串行化，不同 app 互不影响。
  4. 导入场景允许显式传入 record_number，同时同步提升 counter，避免后续回退冲突。
