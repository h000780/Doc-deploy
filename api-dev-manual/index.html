Nata API 设计规范手册

内部开发参考文档

概述与核心原则

本文档定义了 Nata 项目中后端 API 接口的设计规范。所有新增和维护的 API 均需遵循此标准，以确保系统的一致性、可维护性和易用性。

核心原则

RESTful 优先

资源导向设计，URL 代表「资源」，HTTP Method 代表「动作」

语义清晰

接口路径和参数应具有明确的业务含义，优先使用英语名词

扁平化与关联

强依赖用嵌套，弱关联用打平；不要为了表达关系而无限加深 URL 路径

一致性

整个系统的分页、筛选、错误码格式必须完全统一

与业界产品对比

产品API 风格示例
Kintone查询参数传资源IDGET /k/v1/records.json?app=123
Airtable路径嵌套GET /v0/{baseId}/{tableId}
Notion扁平 + 过滤POST /v1/databases/{id}/query
Nata路径嵌套GET /api/apps/{appId}/records

Private API vs Public API

Private API 仅支持内部客户端通过 Cookie (Session) 认证； 而 Public API 支持全场景混合认证。外部系统主要使用 API Token (Bearer Token)，而内部 JS SDK 则可灵活选择 Cookie 或 API Token。

特征Private API (非公開)Public API (公開)
受众官方客户端 (PC/Mobile/MP)第三方系统、集成脚本、内部 JS SDK
稳定性可能频繁变更极端注重兼容性
版本控制通常不严格区分版本强制版本化 (/v1/, /v2/)
限流策略较宽松严格限流 (基于 App 与 Token)
认证方式仅 Cookie (Session)Cookie (Session) 或 API Token
路径前缀/api/.../api/v1/...

架构优势：底层统一

虽然访问层物理隔离，但 Private 与 Public API 共享同一套 Service 层逻辑。这意味着业务更新双侧同步生效，且内部接口公开化只需增加路由与 DTO 过滤，极大降低维护成本并确保行为一致。
